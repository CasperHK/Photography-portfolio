# AGENTS.md

This file is the project playbook for coding agents. Follow it to make safe, predictable changes with minimal regressions.

## 1. Repository Layout

- Monorepo root contains orchestration files only.
- Backend lives in `src/backend`.
- Frontend lives in `src/frontend`.
- Do not assume old root-level `backend` or `frontend` paths.

## 2. Tech Stack

- Frontend: Solid + Rsbuild + TanStack Router file-based routing.
- Backend: Go + chi + SQLite.
- Infra: Docker Compose.

## 3. Frontend Routing Rules

- Route files are in `src/frontend/src/routes`.
- Page components are in `src/frontend/src/pages`.
- Shared frontend constants/types should live in small reusable modules in `src/frontend/src/pages` or other focused folders.
- Keep `src/frontend/src/App.tsx` as compatibility-only unless explicitly asked to make it an app shell again.
- New pages must include a route file and use TanStack Router conventions.

## 4. Frontend Build Conventions

- `src/frontend/src/routeTree.gen.ts` is generated by TanStack Router plugin.
- Never hand-edit `routeTree.gen.ts`.
- Prefer editing route files under `src/frontend/src/routes` and let generation happen on build/dev.
- Do not manually edit artifacts in `src/frontend/dist`.

## 5. Run Commands (Local)

- Frontend dev:
	- `cd src/frontend`
	- `pnpm run dev`
- Frontend build check:
	- `cd src/frontend`
	- `pnpm run build`
- Backend run:
	- `cd src/backend`
	- `go run .`

## 6. Change Workflow for Agents

- Read target files first.
- Make smallest change that satisfies request.
- Preserve existing visual language and interaction patterns unless user asks for redesign.
- After frontend edits, run `pnpm run build` in `src/frontend`.
- After backend edits, run `go run .` in `src/backend` when feasible.
- Report what changed and how it was validated.

## 7. Common Mistakes to Avoid

- Using wrong paths (`backend` vs `src/backend`, `frontend` vs `src/frontend`).
- Adding routes in nav but forgetting route files.
- Editing generated files directly.
- Leaving unused imports/types after refactor.
- Mixing unrelated refactors into one change.

## 8. Code Quality Guidelines

- Prefer clear, explicit code over clever shortcuts.
- Keep functions focused and composable.
- Reuse shared helpers for API/media/demo logic instead of duplicating.
- Keep TypeScript types close to domain concepts.
- Use concise comments only where intent is non-obvious.

## 9. Delivery Checklist

- Change compiles/builds.
- Route links resolve correctly.
- No generated files were manually modified.
- Summary includes files touched, behavior impact, and verification steps.
